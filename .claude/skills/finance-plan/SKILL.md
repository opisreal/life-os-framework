是 Life OS `05_finance/` 模块的**路径调整规划**入口。
按需触发，非周期。输出按可执行度排序的调整建议清单。

## 前置读取
执行前必读：
1. `05_finance/_design/2026-05-24-finance-module-design.md` §3.3（你的铁律）
2. 最近 3 份 reviews（按时间倒序，monthly / quarterly 都读）
3. `05_finance/goals/pyramid.md` + `goals/active/` 当前目标状态
4. `05_finance/budget.md` 当前预算基线
5. 最近 3 个月 snapshot + spending

## 输入
- `/finance plan` → 不带参数，自动从最近 review 推断诊断焦点（选最严重的问题）
- `/finance plan <自由描述>` → 用户指定问题（如 `/finance plan 储蓄率太低`）
- 也可由 review 末尾的提示触发

## 执行步骤

1. **诊断焦点确定**：
   - 用户有自由描述 → 以此为焦点
   - 没有 → 从最近 review 自动选最严重问题（连续 ⚠️/❌ 的目标、长期超 budget 的类别、储蓄率持续偏低）
   - **如果焦点模糊 → 反问用户确认**，不允许自己拍

2. **生成建议**（按"立即可做 / 短期 / 中期"三档列）：
   - **立即可做（< 1 周）**：取消订阅、调整自动转账金额、修改某账户配置等
   - **短期（1-3 月）**：协商利率、调整 budget 上限、改变消费习惯需要时间
   - **中期（3-12 月）**：副业增加、换工作、卖房等大动作

3. **量化影响预期**（每条建议必须给）：
   - 节省 X 元 / 月（基于最近 3 月 spending 数据外推）
   - 时间线提前 / 延后 Y 月（基于 goal 的月均所需）
   - **如果数据不够无法量化** → 必须标"待验证（数据不足）"，不允许编数字

4. **写入文件**：
   - 路径：`plans/<date>-plan-<topic-slug>.md`（date 用 ISO 日期 YYYY-MM-DD，topic-slug 用拼音或英文）
   - 按设计文档 §2.3 plans schema 写

5. **后续询问**（一次性问完，让用户回答）：
   - 要不要更新 `budget.md`？（如建议把柔性消费 budget 从 20% 压到 17%）
   - 要不要在某个 goal 文件里加调整历史条目？

6. **更新 .manifest.json** 的 `plans` 数组

7. git commit: `feat(finance): plan <topic>`

## 报告模板

````markdown
---
type: finance-plan
created: YYYY-MM-DD
trigger: （触发原因，如"季度复盘发现储蓄率连续 3 月低于 25%"或"用户主动 /finance plan"）
related_goals: [目标 1, 目标 2]
---

# 调整路径 · <主题>

## 当前问题
（基于哪些证据/数据，引用具体 review 文件或 spending/snapshot 数据点）

## 调整建议（按可执行度排序）

### 立即可做（< 1 周）
- [ ] 动作 1 — 预期影响：节省 X/月
- [ ] 动作 2 — 预期影响：节省 Y/月

### 短期（1-3 月）
- [ ] 动作 3 — 预期影响：...

### 中期（3-12 月）
- [ ] 动作 4 — 预期影响：...

## 影响预期
- 月储蓄从 X → Y（达标 Z%）
- 目标 A 达成日期从 ... → ...
- 目标 B 时间线 ...
（每个数字必须给出推导依据）

## 假设条件
- 假设 1：（如"工资保持不变"）
- 假设 2：（如"加密资产价格不发生大波动"）

## 后续跟踪
- 下次月度复盘检查执行情况
- 3 个月后回看本计划，归档或更新
````

## 铁律
- ❌ 不推荐具体投资标的（"买 XX 基金/XX 股票"）—— 只评配置类型与比例
- ❌ 不给税务 / 法律 / 保险 buy 决策的最终建议 —— 只做提示性参考
- ❌ 不编造影响预期数字（必须基于历史数据外推；数据不足则标"待验证"）
- ❌ 不删旧 plan（按日期归档便于回看）
- ❌ 不允许笼统的"节流增收"套话（AI slop）—— 每条建议必须具体到"取消 X 订阅"、"改吃食堂"这种动作级
- ❌ 每条建议必须有"可执行度"标记（立即/短期/中期）
- ✅ 影响预期可以乐观但必须可解释（说明假设条件）
- ✅ 后续询问"要不要更新 budget / goal 调整历史"一次性问完，等用户回答再操作
- ✅ git commit 信息按设计文档约定（`feat(finance): plan <topic>`）

## 输出
执行完毕后简要汇报：
- 写了哪个 plan 文件
- 几条立即/短期/中期建议
- 影响预期摘要（如"月储蓄 +4000，应急金达成提前 5 月"）
- 是否更新了 budget / 哪个 goal 的调整历史
- git commit hash
